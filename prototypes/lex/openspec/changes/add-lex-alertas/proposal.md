# Add lex-alertas — Compliance alerts module

## Why

Per `discoveries/lex-alertas-discovery.md` (REQ-52, SENT TO DEV), Lex incorporates a top-level Alertas module — `/alertas` — that hosts every compliance alert generated by the platform. v1 ships with one alert type, `SCREENING_BLACKLIST_MATCH`, generated when an OPS movement's `counterparty_tax_number` matches a CUIT in the Lex Blacklist (per `lex-blacklist`). The architecture MUST support adding new types (e.g. `DOCUMENT_DUE_DATE`, `LIMIT_OVERAGE`) without restructuring the page or its state machine.

The legacy frontend has no Alertas module — the prototype ships REQ-33 patterns; this change brings them into the new project as a typed, role-gated, state-machined surface. The post-2026-04-24 PM iteration locks the **two-tab layout (Nuevas / Histórico)**, the assignment-driven `new → in_review` transition with timeline tracking, the closure flow with mandatory comment captured atomically with the status change (one `PATCH`, not two requests), the role-aware permissions matrix (ADMIN_LEX full, COMMERCIAL_LEX comment-only on assigned clients, VIEWER_LEX read-only on own assignations, plus a `COMPLIANCE` external role), the Sidebar badge that decrements as alerts close, and the type-extensible registry pattern.

The OPS-side detection engine and the OPS → Lex event channel are out of scope for this frontend spec; per `discoveries/lex-discovery.md` G-11, the technical channel is owned elsewhere.

## What Changes

- Create the `lex-alertas` capability. New spec at `openspec/specs/lex-alertas/spec.md` (materialised via archive) with 10 requirements covering: (a) top-level Sidebar entry with active-count badge; (b) two-tab layout (Nuevas card-list / Histórico table) gated by status; (c) state machine `new → in_review → resolved|dismissed` enforced by the UI; (d) closing transitions capture a mandatory comment in one atomic `PATCH /alert/:id`; (e) assignment flow with state migration between tabs and timeline tracking; (f) detail view with match payload + assignment + combined timeline; (g) Histórico filter set covering tipo, estado, responsable, cliente, fechas, CUIT contraparte; (h) permissions per the role matrix; (i) comments immutable, max 2000 chars, optimistic prepend; (j) type-extensible registry pattern.
- Define the typed surface. `src/pages/Alertas.vue`, `src/lex/alertas/types.ts` (`Alert`, `AlertType`, `AlertStatus`, `AlertTimelineEvent`), `src/lex/alertas/registry.ts` (`AlertTypeRegistry` + `registerAlertType()`), `src/lex/alertas/api.ts` (`GET /alert`, `PATCH /alert/:id`, `POST /alert/:id/comment`, the count endpoint), the card-list and table components, the detail view at `/alertas/:id`, the closure modal.
- Integrate with sibling capabilities — referenced, not edited:
  - `core-layout` — L1/L2/L3 with the segmenter in the L1 actions slot.
  - `core-data-tables` — Histórico table primitive with the canonical column order, period filter primitive.
  - `core-modals` — closure modal pair (Cancelar / Marcar como revisada or Cancelar / Descartar).
  - `core-actions-menu` — per-row Acciones in Histórico.
  - `core-error-handling` — `devWarn` for unknown alert types; toast on failed comment submission.
  - `core-modulo-genericos` — Drawer/timeline pattern reused for the detail view.
  - `core-theming` — `--badge-alerta-*` tokens for status badges.
  - `lex-blacklist` — origin of `SCREENING_BLACKLIST_MATCH` matches.
  - `lex-cliente-detalle` — destination of the `Ver legajo` link in the detail view.
  - `lex-roles` — permissions matrix with the `COMPLIANCE` external role; `useLexRole()` consumed.
